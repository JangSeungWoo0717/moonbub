# 논증 설계소 (ArguMap) — 설계 명세서

- **작성일**: 2026-06-20
- **과목 / 학년**: 국어 / 고1
- **주제**: 논리적으로 소통하는 힘
- **목표 학과**: 산업공학과 (학생부종합전형 세특 활용)
- **배포**: GitHub (Git Publish) → GitHub Pages
- **기술 스택**: 순수 HTML/CSS/JS (빌드 없음), Chart.js(CDN), localStorage

---

## 1. 개요 (Overview)

토론·논설문의 논증을 **분해 → 시각화 → 점수화 → 결함 점검 → 개선 후 재측정**하는
정적 웹 도구. "소통을 측정·개선 가능한 시스템으로 다룬다"는 산업공학적 관점을
국어 논증 학습에 결합한다.

### 핵심 서사 (세특·입시 차별점)
- **국어 역량**: 툴민(Toulmin) 논증 모델로 주장의 논리 구조를 정밀하게 분해·평가.
- **산업공학 역량**: 논증을 *품질관리 대상*으로 보고 평가 지표(KPI)를 직접 설계,
  **PDCA(Plan-Do-Check-Act)** 사이클로 반복 개선. 정량화·시스템적 사고·프로세스
  개선을 한 산출물로 증빙.

### 성공 기준 (Success Criteria)
1. 사용자가 논증을 6요소로 입력하면 구조 트리가 자동으로 그려진다.
2. 5개 지표 기반 종합 건강도 점수(0–100)와 레이더 차트가 표시된다.
3. 규칙 기반으로 주요 논리적 오류를 자동 경고하고, 수동 체크리스트를 제공한다.
4. 버전을 저장하고 개선 전·후 점수를 비교할 수 있다.
5. 모바일·데스크톱 모두에서 깨지지 않는 반응형 UI.
6. GitHub Pages에서 별도 서버·API 키 없이 동작한다.

## 2. 사용자 흐름 (PDCA 1바퀴)

1. **입력 (Plan)** — 툴민 논증 모델 6요소를 칸에 분해 입력
   `주장(Claim) / 근거(Grounds, 복수) / 전제·보강(Warrant·Backing) / 한정(Qualifier) / 반례·반론(Rebuttal)`
2. **시각화 (Do)** — 입력을 논증 구조 트리 다이어그램으로 자동 렌더링
3. **측정 (Check)** — 5개 지표 점수화 → 레이더 차트 + 종합 건강도 점수(0–100)
4. **점검 (Check)** — 논리적 오류 체크리스트(규칙 기반 자동 경고 + 수동 점검)
5. **개선 (Act)** — 초안을 저장하고 수정안과 점수를 전·후 비교

---

## 3. 화면 구성 (모바일 우선, 단일 페이지 + 탭)

- **① 입력 탭** — 논증 6요소 폼, 근거 동적 추가/삭제 버튼, 예시 불러오기.
- **② 분석 탭** — 트리 다이어그램 · 레이더 차트 · 종합점수 게이지 · 오류 경고 리스트.
- **③ 비교 탭** — 저장된 버전 목록, 선택한 두 버전의 전·후 점수 비교 막대그래프.
- **④ 방법론 탭** — 툴민 모델 설명, 점수 산정 기준 공개, 산업공학(품질관리·PDCA·KPI)
  연계 설명. 세특 발표 시 논리적 근거 자료로 사용.

탭 전환은 JS로 섹션 show/hide. 모바일에서는 하단/상단 탭바, 단일 컬럼.

## 4. 데이터 모델

```js
// 하나의 논증(버전)
Argument = {
  id: string,            // 고유 id (타임스탬프 기반)
  title: string,         // 사용자가 붙인 이름 (예: "사형제 폐지 - 초안")
  createdAt: number,     // 저장 시각
  claim: string,         // 주장
  grounds: string[],     // 근거(복수)
  warrant: string,       // 전제·보강
  qualifier: string,     // 한정 (예: "대체로", "특정 조건에서")
  rebuttal: string,      // 반례·반론 및 그에 대한 대응
  scores: ScoreResult,   // 측정 결과 (저장 시점 스냅샷)
  fallacies: string[]    // 감지/체크된 오류 키 목록
}

ScoreResult = {
  groundsSufficiency: number, // 0-20
  logicalConnection: number,  // 0-20
  counterHandling: number,    // 0-20
  claimClarity: number,       // 0-20
  fallacyCleanliness: number, // 0-20
  total: number               // 0-100
}
```

저장은 `localStorage` 키 하나(`argumap_versions`)에 `Argument[]` JSON으로 보관.

## 5. 점수 산정 로직 (학생이 설계한 KPI — 방법론 탭에 투명 공개)

5개 차원, 각 0–20점 → 합산 0–100. 모든 규칙은 클라이언트 측 결정론적 계산.

| 지표 | 측정 방식 (규칙 기반) |
|---|---|
| **근거 충분성** (groundsSufficiency) | 근거 개수(2개 이상 가점) + 구체성: 수치/단위/출처 키워드(`%`, `명`, `년`, `통계`, `자료`, `연구`, `에 따르면`) 탐지 |
| **논리 연결성** (logicalConnection) | 전제·보강(warrant) 작성 여부 및 길이. 주장-근거를 잇는 설명 존재 시 가점 |
| **반론 대응력** (counterHandling) | 반례·반론(rebuttal) 입력 여부 + 대응 표현(`그러나`, `하지만`, `반박`, `그럼에도`) 탐지 |
| **주장 명료성** (claimClarity) | 한정어 적절성, 주장 길이(너무 짧거나 길면 감점), 모호어(`등등`, `뭔가`, `아마`, `것 같다`) 감점 |
| **오류 청결도** (fallacyCleanliness) | 오류 점검에서 감지된 오류 1건당 감점 (기본 20점에서 차감) |

각 지표 계산 함수는 입력 텍스트와 임계값만으로 점수를 반환한다(테스트 가능).

## 6. 오류 점검 규칙 (규칙 기반 자동 + 수동 확인)

대상 오류와 1차 자동 탐지 신호:

| 오류 | 자동 탐지 신호(예시 패턴) |
|---|---|
| 순환논법 | 주장의 핵심어가 근거에 거의 그대로 반복 |
| 성급한 일반화 | `모두`, `항상`, `절대`, `누구나`, `예외 없이` + 근거 부족 |
| 흑백논리 | `~아니면`, `둘 중 하나`, `오직 ~만` 패턴 |
| 허수아비 | 반론 인용이 과장/극단 표현과 결합 |
| 인신공격 | `그 사람은`, `너는`, 인격 비하어 |
| 권위에 호소 | `유명한`, `전문가가 그랬으니` + 근거 부재 |

자동 탐지는 "의심" 경고로만 표시하고, 학생이 체크박스로 최종 판단(수동 확인).
규칙은 `fallacy.js`에 규칙 객체 배열로 분리해 추가/수정 용이하게 한다.

## 7. 파일 구조

```
index.html                      # 단일 페이지, 4개 탭 섹션
styles/
  main.css                      # 기본 스타일·레이아웃·테마
  responsive.css                # 모바일 우선 미디어쿼리
js/
  model.js                      # 데이터 모델, 기본값, id 생성
  scoring.js                    # 5개 지표 점수 계산 (순수 함수)
  fallacy.js                    # 오류 탐지 규칙 + 검사 함수
  tree.js                       # 논증 구조 트리 다이어그램 렌더
  charts.js                     # Chart.js 레이더/비교 차트
  storage.js                    # localStorage 저장/불러오기
  app.js                        # 탭 제어, 폼 이벤트, 전체 조립
data/
  sample-arguments.json         # 예시 논증 1~2개
.github/workflows/deploy.yml    # GitHub Pages 자동 배포
README.md                       # 프로젝트 소개 + 세특 서사 + 사용법
```

각 JS 모듈은 ES Module(`<script type="module">`)로 분리, 단일 책임 유지.
`scoring.js`/`fallacy.js`는 DOM 비의존 순수 함수로 만들어 테스트·재사용 가능.

## 8. 반응형 / 접근성

- 모바일 우선 CSS. 단일 컬럼(모바일) ↔ 2컬럼(데스크톱, 입력+분석 동시).
- 터치 친화적 버튼 크기(최소 44px), 가독성 높은 폰트·대비.
- 차트는 컨테이너 폭에 맞춰 리사이즈.
- 시맨틱 태그, 라벨-인풋 연결, 키보드 포커스 가시화.

## 9. 배포 (GitHub Pages)

- 빌드 없음 → 정적 파일 그대로 서빙.
- `.github/workflows/deploy.yml`로 `main` push 시 Pages 자동 배포.
- 서버·환경변수·API 키 불필요.
- Git Publish로 새 저장소 생성 후 push.
